# CampusHub Phase 4 项目广告与校园展示升级设计

日期：2026-05-21

## 1. 目标与边界

Phase 4 的目标是把项目广告从静态信息流升级为可内测的校园展示与协作入口。它服务四类场景：项目组队、作品展示、社团招募、校园活动宣传。CampusHub 负责结构化发布、内容审核、标签检索、过期管理、联系方式门槛、收藏评论、举报治理、精选展示和运营后台可见性。

本阶段不把项目广告做成交易闭环，不做交易本金托管，不做平台内佣金分账，不做实时聊天，不做复杂推荐算法。项目发起人与参与者仍通过微信/QQ、线下或外部链接继续沟通；平台沉淀可信发布、内容质量、互动记录、举报处理和校园展示页面。

## 2. 推荐路线

采用“发布审核 + 展示详情 + 互动治理 + 运营精选”的路线，而不是直接建设复杂社区流或算法推荐。

- 发布审核：注册用户可提交项目广告，默认进入 `PENDING_REVIEW`；管理员审核通过后公开展示。
- 展示详情：列表页面向快速浏览，详情页承载项目描述、招募需求、作品链接、标签、附件、联系方式展示规则、收藏评论和举报。
- 校园展示：精选位由运营手动设置，支持首页或项目广告页突出展示优质项目、活动和作品。
- 生命周期：项目广告必须有过期时间；到期后不再出现在公开列表，但详情和后台记录保留。
- 治理闭环：项目广告、评论和发布者均可被举报；后台可查看待审、精选、过期、举报和互动数据。

## 3. 用户与权限规则

### 3.1 发布者

普通注册用户即可提交项目广告，不要求额外保证金。原因是项目广告不直接承载交易本金和履约责任，风险低于二手交易发布和店铺服务。若后续出现大量垃圾内容，可再引入信用分、发布频率或人工白名单限制。

发布者可以编辑自己的草稿、待审或被拒广告；已通过审核的广告再次修改关键字段后应回到待审状态。发布者可以主动下架自己的广告。后端必须校验发布者身份，前端只做操作入口提示。

### 3.2 浏览者

普通用户可浏览公开项目广告、查看详情、收藏、评论和举报。联系方式默认不在列表页公开；详情页可按规则展示：公开展示、登录后展示、评论/收藏后展示，或仅显示“通过外部链接/站内评论联系”。第一版推荐使用 `LOGIN_ONLY` 作为默认联系方式展示规则。

### 3.3 管理员

管理员可以审核通过、拒绝、设为精选、取消精选、标记违规下架、查看过期项目和处理举报。复杂自动风控、机器审核和个性化推荐不进入本阶段。

## 4. 项目广告模型

现有 `project_ads` 表只有发布者、标题、描述、链接、联系方式、状态、浏览量和时间字段。Phase 4 通过新 Flyway 迁移 V9 扩展，不能修改已应用 V1-V8。

建议新增字段：

- `ad_type`：广告类型，建议值 `TEAM_UP`、`PORTFOLIO`、`CLUB_RECRUITMENT`、`CAMPUS_EVENT`、`OTHER`。
- `summary`：列表摘要。
- `tags`：逗号分隔标签，第一版不单独建标签表，避免过度设计。
- `campus_zone`：校区范围，复用校区优先策略；可为空或 `OTHER`。
- `cover_file_id`：封面文件，复用文件资源。
- `contact_visibility`：联系方式展示规则，建议值 `PUBLIC`、`LOGIN_ONLY`、`INTERACTION_ONLY`、`HIDDEN`。
- `expires_at`：过期时间。
- `featured`：是否精选。
- `featured_priority`：精选排序权重。
- `review_note`：审核意见。
- `reviewed_by` / `reviewed_at`：审核人和时间。
- `published_at`：首次公开时间。
- `closed_at`：发布者下架或管理员处理时间。

状态建议：

- `DRAFT`：草稿。
- `PENDING_REVIEW`：待审核。
- `APPROVED`：已公开。
- `REJECTED`：审核拒绝。
- `EXPIRED`：已过期。
- `CLOSED`：发布者下架。
- `BLOCKED`：平台违规下架。

第一版过期可以由查询逻辑过滤 `expires_at <= now`，并提供后台“过期”视图；不强制引入定时任务。后续若需要可添加低频定时任务把状态批量更新为 `EXPIRED`。

## 5. 详情与联系方式展示

项目广告列表不展示完整联系方式，只展示类型、标题、摘要、标签、发布者、浏览量、收藏数、评论数、过期时间和精选标识。

详情页展示完整描述、招募需求、作品/活动链接、附件图片、评论、收藏按钮、举报入口和联系方式提示。联系方式展示规则：

1. 发布者本人和管理员始终可见。
2. `PUBLIC`：详情页公开展示。
3. `LOGIN_ONLY`：登录用户可见；未登录用户看到登录提示。
4. `INTERACTION_ONLY`：收藏或评论后可见，用于减少爬取和无意向骚扰。
5. `HIDDEN`：不展示联系方式，只保留外部链接、评论或线下说明。

若联系方式来自用户注册时的 WeChat/QQ，可在创建时生成文本快照，避免后续用户修改联系方式导致历史广告语义变化。第一版允许发布者填写广告专用联系方式，但前端应提示不要填写敏感隐私信息。

## 6. 互动、附件与治理复用

Phase 4 应复用现有通用能力，而不是为项目广告重复建表：

- 文件附件：复用 `file_resources` 与 `file_bindings(target_type = PROJECT_AD, target_id = id)`。
- 收藏与评论：复用 interaction target model，`target_type = PROJECT_AD`。
- 举报：复用 moderation reports，`target_type = PROJECT_AD`。
- 审核记录：复用或补充现有 review/moderation 记录；项目广告自身仍保留最近审核状态和意见，便于列表查询。
- 站内通知：审核通过、审核拒绝、精选、违规下架和评论提醒应创建通知。

评论不做楼中楼、富文本和实时刷新。附件第一版只展示已绑定图片/文件，不做复杂素材管理器。

## 7. 前端体验设计

### 7.1 项目广告列表页 `/project-ads`

现有时间线升级为校园展示市场：

- 顶部筛选：类型、标签关键词、校区、是否精选、是否未过期。
- 顶部行动：发布项目广告、刷新。
- 卡片展示：封面、类型、标题、摘要、标签、发布者、浏览量、收藏数、评论数、过期时间、精选标识。
- 空状态：提示“发布你的项目、作品或社团招募”。
- 移动端：卡片单列、筛选折叠或自然换行，无横向溢出。

### 7.2 项目广告详情页 `/project-ads/:id`

详情页包含：

- 项目头图、标题、类型、精选标识、状态提示。
- 完整描述、标签、校区、外部链接、过期时间。
- 联系方式区域，按 `contact_visibility` 显示内容或提示。
- 附件/图片区域。
- 收藏、评论、举报入口。
- 发布者本人操作：编辑、提交审核、下架。
- 管理员/运营操作入口可在后台完成，详情页不必承载完整审核台。

### 7.3 发布与管理

第一版可在 `/project-ads` 使用抽屉/对话框发布，也可新增 `/project-ads/manage` 管理页。推荐新增轻量管理页，避免列表页过重：

- 我的项目广告列表：草稿、待审、已发布、被拒、过期、下架。
- 创建/编辑表单：类型、标题、摘要、描述、标签、校区、链接、联系方式、联系方式展示规则、过期时间。
- 提交审核、编辑、下架操作。

### 7.4 运营后台

在 `/admin/ops` 增加“项目广告”标签页：

- 待审核列表。
- 已精选列表。
- 即将过期或已过期列表。
- 违规/举报入口链接。
- 审核通过、拒绝、精选、取消精选、平台下架操作。

复杂图表不进入第一版；先保证数据可查、状态可改、操作可追溯。

## 8. 后端 API 设计

建议新增或扩展：

- `GET /api/project-ads`：公开列表，支持类型、标签、校区、精选、关键词过滤，只返回公开且未过期内容。
- `GET /api/project-ads/featured`：精选项目广告。
- `GET /api/project-ads/{id}`：详情，增加浏览量并按 viewerId 决定联系方式可见性。
- `GET /api/project-ads/users/{userId}`：发布者管理列表。
- `POST /api/project-ads?publisherId=`：创建草稿或待审广告。
- `PUT /api/project-ads/{id}?publisherId=`：发布者编辑。
- `POST /api/project-ads/{id}/submit?publisherId=`：提交审核。
- `POST /api/project-ads/{id}/close?publisherId=`：发布者下架。
- `GET /api/admin/ops/project-ads`：后台监控列表，支持状态筛选。
- `POST /api/admin/ops/project-ads/{id}/approve?adminId=`：审核通过。
- `POST /api/admin/ops/project-ads/{id}/reject?adminId=`：审核拒绝。
- `POST /api/admin/ops/project-ads/{id}/feature?adminId=`：设为精选。
- `POST /api/admin/ops/project-ads/{id}/unfeature?adminId=`：取消精选。
- `POST /api/admin/ops/project-ads/{id}/block?adminId=`：平台下架。

路径可结合现有控制器微调，但用户侧、发布者管理和运营后台职责应分清。

## 9. 测试与验证

后端测试重点：

- 创建项目广告默认进入待审或草稿状态，未审核不出现在公开列表。
- 审核通过后公开列表可见，并记录 `published_at` 和审核信息。
- 过期项目不出现在公开列表。
- 发布者本人可编辑/下架自己的广告，不能操作他人广告。
- 详情访问增加浏览量，并按联系方式展示规则返回 contactVisible/contactInfo。
- 精选项目按 `featured_priority` 排序。
- 审核拒绝、通过、精选、下架会写入站内通知。

前端验证重点：

- `/project-ads` 列表筛选、精选展示、卡片和空状态可用。
- `/project-ads/:id` 详情页显示描述、标签、附件、互动入口和联系方式规则。
- 发布/编辑/提交审核/下架流程可用。
- `/admin/ops` 项目广告标签页能审核、精选、下架。
- 桌面和移动端无明显横向溢出。
- `npm --prefix frontend run build` 通过，仅允许既有 Element Plus chunk-size 警告。

## 10. 非目标

Phase 4 不做：

- 项目融资、赞助、交易本金托管；
- 每条广告保证金或押金冻结；
- 平台内实时聊天；
- 复杂推荐算法或信息流排序模型；
- 楼中楼评论、私信、关注系统；
- 多学校开放；
- 小程序客户端；
- 自动爬取外部作品集内容。

这些能力可在项目广告真实使用数据积累后再评估，不应阻塞校园展示主流程上线。

## 11. 自检结论

本设计聚焦单一 Phase 4 子项目，范围可由一个实施计划完成；保持 CampusHub 支付边界，不读取或保存支付宝密钥，不引入本金托管；数据库变更必须通过 V9+ 新迁移完成；联系方式沿用注册联系与展示门槛思路；附件、收藏、评论、举报和通知复用现有通用能力；推荐位采用运营手动精选而非复杂算法。
