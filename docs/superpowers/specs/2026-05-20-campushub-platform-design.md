# CampusHub 平台发展总设计

日期：2026-05-20

## 1. 背景与定位

CampusHub 的目标不再是课程演示原型，而是逐步发展为面向真实校园场景的生活服务与交易平台。平台的核心价值不是替代微信、QQ 或线下沟通，而是把校园内零散需求结构化：需求发布、可信匹配、履约记录、信用沉淀、举报纠纷、身份保证金、运营数据和服务费收取。

平台第一阶段以校园跑腿/代取作为主业务。二手交易、学生店铺和项目广告保留为后续业务线，但不与跑腿第一阶段抢资源。平台能力以业务需要驱动沉淀：身份、联系方式、保证金、通知、信用、后台、点位数据、服务费、支付边界都先服务跑腿，后续复用于二手和店铺。

## 2. 总路线

CampusHub 采用“跑腿先行，按业务线逐步铺开”的路线。

1. 校园跑腿/代取：快递代取、外卖/食堂代取、打印代取、临时帮忙等高频需求。
2. 二手交易：商品发布、浏览、联系、评价、举报和信用展示。
3. 学生店铺：技能服务、预约档期、店铺页面、服务项目、商家后台和评价。
4. 项目广告：项目组队、作品展示、社团招募、校内活动宣传。
5. 治理与运营后台：随第一阶段同步建设，逐步增强数据分析和风控。

开放节奏采用阶段开放：Alpha 小范围内测，Beta 种子用户，Campus 全校开放。当前不以某个单一成功指标作为立即判定标准，而是先完成子项目和阶段路线，再按阶段公测和评估。业务线优先级按跑腿、二手、店铺、项目广告组织，平台能力在各业务阶段中抽象沉淀。

## 3. 用户身份、联系方式与保证金

### 3.1 注册要求

所有用户注册时必须完成学校邮箱验证，并提供微信或 QQ 联系方式，至少填写一种。联系方式用于交易达成后的业务沟通，不在公开列表中直接暴露，避免骚扰和爬取。

注册时用户需要接受平台服务协议、交易风险提示和隐私说明。学校邮箱验证是第一层身份可信基础，后续低风险身份申请不重复人工审核。

### 3.2 身份类型

- 普通用户：可浏览、发布跑腿需求、收藏、评论、评价和举报，不需要保证金。
- 跑腿接单者：支付 5 元身份保证金后自动开通，可抢单或申请接单。
- 二手交易发布者：支付 10 元身份保证金后自动开通，可发布二手商品。
- 学生店铺商家：支付 20 元身份保证金后进入人工审核，通过后可创建店铺和服务项目。

### 3.3 保证金边界

保证金是身份准入和违规约束工具，不是每笔订单的押金。CampusHub 第一阶段不做逐单保证金冻结/解冻，也不托管每笔交易本金。

后续可支持严重违规扣除保证金、保证金不足时暂停身份、保证金退还申请、无未完成订单/纠纷检查、保证金流水审计。

## 4. 收费与资金边界

CampusHub 只收平台服务费和身份保证金，不提供交易本金托管、转存佣金或每笔订单资金保管。

跑腿任务中：

- 抢单模式只收基础服务费。
- 申请模式收基础服务费，并预留额外费用规则。
- 服务费定价在本设计中不固定；实现时应建立可配置的费用策略表或配置项，使运营者后续能调整基础服务费、申请模式附加费和豁免规则。

二手交易与学生店铺后续也遵循同一原则：平台负责发现、信用、流程和治理，不默认托管买卖双方线下或微信/QQ 转账资金。

支付能力应保持可扩展。当前 CampusHub 不直接接触支付宝私钥，生产支付可继续通过 API-Transfer-Station 支付中心完成服务费和保证金收款。

## 5. 跑腿任务核心机制

### 5.1 任务类型

第一阶段跑腿任务覆盖：

- 快递代取；
- 食堂/外卖代取；
- 打印/资料代取；
- 校内临时帮忙；
- 其他自定义任务。

任务发布必须包含标题、任务类型、起点校区、终点校区、地点文字描述、截止时间、报酬、联系方式展示规则、接单模式、完成确认方式和补充备注。

### 5.2 接单模式

任务发布者在发布时选择接单模式。

抢单模式适合低风险、时间敏感、标准化任务。任务进入任务大厅后，符合条件的接单者可先到先得。任务卡片应突出起终点、校区、报酬、截止时间、预计耗时、发布者信用和基础服务费提示。

申请模式适合高风险、高金额、敏感或需要筛选人的任务。接单者提交申请说明，发布者选择一个申请人。申请模式预留额外收费规则，但第一阶段不固定价格。

### 5.3 履约状态机

跑腿任务状态应清晰可追踪：

1. 草稿；
2. 已发布；
3. 待接单或待选择申请；
4. 已接单；
5. 前往取件点；
6. 已取件；
7. 配送中；
8. 已送达；
9. 待确认；
10. 已完成；
11. 已取消；
12. 异常处理中；
13. 纠纷处理中。

第一阶段可以在 UI 上合并部分状态，但数据库应保留足够表达能力，避免后续重构。

### 5.4 完成确认

完成确认采用混合模式。

普通低风险任务使用完成码：发布者提供完成码，接单者提交后任务自动完成或进入完成确认。高风险、高金额或敏感任务要求图片凭证和发布者手动确认。系统可根据任务类型、金额、用户信用和历史纠纷推荐确认方式，但发布者仍可在允许范围内选择。

### 5.5 异常与纠纷

接单后应提供异常上报入口，包括无法取件、地点错误、联系不上、超时、物品异常、对方取消等。异常进入后台任务监控和纠纷处理队列。所有关键操作写入任务事件日志，便于客服/管理员判断责任。

## 6. 校区位置与地图策略

第一阶段不直接建设精细 POI 和距离计价，也不依赖离线高德地图数据。位置系统先采用校区级结构化字段和文字详情。

起点/终点校区枚举：中校区、西校区、东校区、北校区、南校区、高新校区、先研院、科学岛、其他。

任务发布时保存：

- 起点校区；
- 终点校区；
- 起点具体地点文字；
- 终点具体地点文字；
- 用户备注；
- 后台可分析的原始文本。

运行一段时间后，根据真实数据统计高频路线和地点，再决定常用 POI、距离估算、基础额度和服务费规则。

高德 API 选择上，若要服务器端计算路线/距离，应优先申请 Web服务 API；若要前端交互地图、选点、拖拽定位，再补 Web端 JS API。第一阶段不把任一地图 API 作为核心依赖。

## 7. 跑腿 UI 体验

跑腿 UI 采用混合方向：任务大厅接近美团/饿了么配送大厅，接单后履约工作台接近京东/达达配送工作台。

### 7.1 任务大厅

任务大厅面向接单者，重点是快速决策。卡片展示：

- 任务类型；
- 起点校区与终点校区；
- 具体地点摘要；
- 报酬；
- 截止时间；
- 接单模式；
- 发布者信用；
- 是否需要图片凭证；
- 是否高风险提示。

筛选维度包括校区、任务类型、报酬区间、截止时间、接单模式和是否仅看抢单。

### 7.2 发布流程

发布流程应简洁，适合手机端填写。优先采用分段表单：任务类型、地点、报酬/截止时间、接单方式、确认方式、联系方式展示和提交确认。

### 7.3 履约工作台

接单后进入工作台，展示当前步骤、下一步操作、联系入口、异常上报、任务详情和事件记录。工作台要减少双方误会，明确“现在该谁做什么”。

## 8. 通知系统

第一阶段只做站内通知。通知类型包括：任务被抢单、申请提交、申请通过/拒绝、任务状态变更、完成码提交、图片凭证提交、异常上报、纠纷处理、评价提醒、身份保证金状态、店铺审核结果。

微信/QQ 是交易沟通渠道，但平台内必须保留关键状态通知和操作入口。邮件、浏览器推送、短信和小程序订阅消息作为后续增强，不阻塞第一阶段。

## 9. 后台运营与治理

后台第一阶段按数据运营后台设计，但分层落地。

必须具备：

- 用户列表；
- 身份与保证金记录；
- 店铺商家审核；
- 任务监控；
- 异常任务；
- 纠纷处理；
- 举报处理；
- 违规记录；
- 黑名单；
- 信用分调整；
- 内容审核队列。

数据运营能力包括：任务发布量、接单转化、平均响应时间、完成率、异常/纠纷数量、服务费统计、活跃用户、热门校区路线、任务类型分布。复杂图表可以后续增强，但第一阶段应保证数据可查、可导出、可追溯。

## 10. 二手交易路线

二手交易是第二业务线。它应复用注册联系方式、二手发布者保证金、信用分、举报、评价和服务费能力。

二手交易不托管交易本金。平台提供商品发布、图片、分类、搜索、收藏、联系、下架、评价、举报和违规处理。二手发布者支付 10 元保证金后自动开通。商品详情页应突出卖家信用、联系方式展示条件、交易风险提示和举报入口。

二手交易的 UI 可参考淘宝/京东的商品信息层级，也参考闲鱼的个人信用和沟通导向。第一目标是可信信息发布和线下/私聊成交辅助，不做复杂购物车和平台结算。

## 11. 学生店铺路线

学生店铺是第三业务线。它适合摄影、贴膜、电脑维护、妆造、家教、设计、剪辑等持续服务。店铺商家支付 20 元保证金后进入人工审核，通过后开店。

店铺功能包括：店铺首页、服务项目、价格区间、服务范围、档期/预约、作品展示、评价、联系方式、举报和商家后台。UI 可参考淘宝/京东店铺机制：店铺门面、服务橱窗、评价、信誉、客服联系，但保持校园轻量化。

预约服务不默认托管本金。平台可收服务费，并记录预约、确认、完成、取消和评价。

## 12. 项目广告路线

项目广告作为第四业务线，用于项目组队、作品展示、社团招募、校园活动宣传。它不作为第一阶段交易主闭环，但可增强平台内容和校园连接价值。

功能包括广告发布、审核、浏览、收藏、评论、联系方式展示、过期时间和举报。后续可支持推荐位、标签、作品集和项目主页。

## 13. 移动端策略

第一阶段先做响应式 Web，保证手机浏览器可用。架构和 API 预留小程序复用能力，但不立即开发微信小程序。

响应式 Web 的重点页面包括：任务大厅、任务发布、履约工作台、我的任务、站内通知、身份申请、钱包/保证金、举报/异常入口。桌面端后台保持管理效率，移动端用户侧保持操作轻量。

## 14. 数据模型方向

后续数据模型应围绕以下核心对象扩展：

- 用户与联系方式；
- 身份申请与角色；
- 保证金账户与保证金流水；
- 跑腿任务；
- 任务申请；
- 任务接单/履约记录；
- 任务事件日志；
- 校区位置字段与原始地点文本；
- 站内通知；
- 异常与纠纷；
- 举报、违规、信用记录；
- 服务费记录；
- 后台运营统计。

所有金额规则应配置化或策略化，不在业务代码中写死长期规则。当前 5/10/20 元身份保证金是初始业务规则，可作为配置或数据库种子记录。

## 15. API 与系统边界

用户侧 API 应支持未来 Web、小程序和可能的移动端复用。不要把业务流程写死在前端。关键状态流转由后端校验，前端只发起动作。

支付边界保持清晰：CampusHub 可调用外部支付中心创建保证金或服务费支付单，接收内部回调后更新本地记录。CampusHub 不读取、不复制、不保存支付宝密钥正文。

站内通知、任务事件日志、保证金流水、服务费流水和管理员操作日志都应可审计。

## 16. 阶段开放计划

### 16.1 Alpha 内测

范围：几十人，熟人、班级或社团。

目标：跑通注册、联系方式、跑腿发布、抢单/申请、履约工作台、完成确认、站内通知、异常上报、后台处理、保证金身份开通。

### 16.2 Beta 种子用户

范围：几百人，覆盖多个宿舍区和学院。

目标：观察供需密度、高频校区路线、任务类型分布、接单速度、异常类型、用户反馈。此阶段开始沉淀 POI 和基础额度规则。

### 16.3 Campus 全校开放

范围：USTC 全校学生。

目标：强化治理、后台、风控、数据运营、客服流程、服务费规则、二手和店铺业务接入。

## 17. 近期子项目分解

1. 跑腿任务 V1：任务发布、抢单/申请、状态机、履约工作台。
2. 身份与保证金 V1：联系方式注册、身份申请、5/10/20 元保证金规则、保证金流水。
3. 校区位置 V1：校区枚举、地点文本、路线数据收集。
4. 站内通知 V1：关键状态通知和通知中心。
5. 运营后台 V1：用户、身份、任务、异常、举报、保证金、基础数据。
6. 移动响应式 V1：任务大厅、发布、工作台、我的任务、通知。
7. 二手交易 V2：发布身份、商品详情、联系、评价、举报。
8. 学生店铺 V3：商家审核、店铺页、服务项目、预约。
9. 项目广告 V4：内容展示、审核、收藏、评论。

## 18. 非目标

第一阶段不做：

- 每笔订单资金托管；
- 每笔订单保证金冻结；
- 实时轨迹；
- 精细地图路径计费；
- 全量多学校开放；
- 微信小程序；
- 复杂动态定价；
- 全自动派单。

这些能力可以作为后续路线保留，但不能阻塞跑腿业务真实落地。
