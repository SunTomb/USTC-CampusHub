# CampusHub Phase 2 二手交易升级设计

日期：2026-05-21

## 1. 目标与边界

Phase 2 的目标是把当前只读展示型二手模块升级为可真实内测的校园二手信息发布与可信联系闭环。平台继续遵守 CampusHub 总边界：不托管交易本金，不做购物车和平台内货款结算，不做逐单保证金冻结。CampusHub 负责身份准入、信息结构化、图片展示、联系门槛、信用沉淀、举报治理、评价记录和服务费接口预留。

本阶段重点不是复制淘宝完整交易系统，而是做一个更接近闲鱼校园版的轻量交易平台：卖家通过二手发布者身份准入后发布商品；买家浏览商品详情、收藏、评论、表达购买意向，并在满足规则后看到卖家联系方式；双方线下或微信/QQ 协调交易；平台保留关键行为、评价、举报、服务费记录和治理入口。

## 2. 推荐路线

采用“可信发布 + 商品详情 + 联系意向”的路线，而不是立即建设完整订单支付系统。

- 发布门槛：只有已通过 `GOODS_PUBLISHER` 身份的用户可以发布二手商品。该身份沿用 Phase 1 身份保证金机制，保证金为 10 元并自动开通。
- 交易主流程：商品发布、审核/上架、浏览、收藏、评论、购买意向、联系方式展示、线下成交、卖家标记售出、买卖双方评价。
- 服务费边界：Phase 2 只预留二手服务费记录和支付中心接口，不强制每笔二手交易线上收费。若运营后续启用服务费，可在购买意向或成交确认节点创建 `ServiceFeeRecord`。
- 治理闭环：商品、评论、用户均可被举报；后台能查看二手举报、商品状态和评价记录。

这一路线最适合当前项目，因为现有 `goods`、`file_bindings`、`report_records`、`favorites`、`comments`、`service_fee_records` 已有部分通用模型，但二手订单和支付还没有真实工作流。先做信息发布与可信联系可以快速形成可用产品，同时避免过早引入本金托管和复杂纠纷。

## 3. 用户与权限规则

### 3.1 二手发布者身份

用户必须先申请 `GOODS_PUBLISHER` 身份，支付 10 元身份保证金后自动通过。后端发布接口必须校验 `role_applications` 中该用户 `role_type = GOODS_PUBLISHER`、`deposit_status = PAID`、`review_status = APPROVED`。

未具备身份的用户在前端点击发布时应看到引导卡片，说明“发布二手商品需要 10 元发布者保证金”，并跳转到身份保证金页。后端仍是最终权限边界，不能只依赖前端隐藏按钮。

### 3.2 买家身份

普通注册用户即可浏览、收藏、评论、提交购买意向、举报和评价。买家不需要额外保证金。后续若发现恶意骚扰，可以通过信用分、黑名单和联系频率限制治理。

### 3.3 管理员能力

管理员可以查看商品列表、举报、评价和异常商品；可以下架违规商品、处理举报、记录违规。Phase 2 后台优先补“可查可处理”，复杂风控自动化放到后续阶段。

## 4. 商品信息模型

现有 `goods` 表包含标题、描述、价格、原价、成色、交易地点、状态、浏览量等基础字段。Phase 2 新增字段应通过新 Flyway 迁移添加，不能修改已应用的 V1-V5。

建议新增：

- `campus_zone`：交易校区，复用 Phase 1 校区枚举思想。
- `contact_visibility`：联系方式展示规则，默认 `INTENT_ONLY`。
- `delivery_method`：交易方式，例如 `OFFLINE_MEETUP`、`SELF_PICKUP`、`NEGOTIATED`。
- `published_at`、`updated_at`、`sold_at`：发布、更新、售出时间。
- `sold_to_user_id`：可选，卖家标记售出时指定买家。
- `service_fee_policy`：预留服务费策略标识，默认 `NONE` 或 `OPTIONAL_ON_SUCCESS`。

商品状态建议统一为：

- `DRAFT`：草稿。
- `PUBLISHED`：已上架。
- `RESERVED`：已预留。
- `SOLD`：已售出。
- `OFF_SHELF`：卖家下架。
- `BLOCKED`：平台下架或风控冻结。

第一版可以不做草稿 UI，但数据库和服务层保留状态表达，便于后续扩展。

## 5. 图片附件设计

复用现有 `FileResource` 与 `FileBinding`：

- `target_type = GOODS`
- `target_id = goods.id`
- `usage_type = COVER` 表示封面图
- `usage_type = GALLERY` 表示详情图
- `sort_order` 控制图片顺序

Phase 2 需要补充文件上传与绑定接口。上传接口接收图片文件，保存文件元数据到 `file_resources`，并把文件放到配置的上传目录。绑定接口将文件绑定到商品。发布商品时可以先创建商品，再上传/绑定图片；也可以先上传临时图片，再在发布成功后绑定。为了降低复杂度，推荐第一版采用“先创建商品、再上传绑定”的流程。

图片限制：每个商品最多 9 张图片；仅允许 JPEG、PNG、WebP；单张大小限制由配置控制，初始建议 5MB；封面默认使用排序第一张图。

前端商品列表显示封面图，详情页显示轮播/九宫格图片。没有图片时显示统一占位。

## 6. 联系方式展示规则

联系方式不应在商品列表公开展示。商品详情页默认只显示卖家昵称、信用分、校区、历史评价概况和“联系卖家”按钮。

Phase 2 推荐规则：

1. 卖家本人查看自己的商品：可看到完整联系方式。
2. 普通买家未提交购买意向：看不到微信/QQ，只能评论、收藏、举报或点击“我想要”。
3. 买家提交购买意向后：系统创建轻量 `GoodsIntent` 记录，买家可查看卖家联系方式快照，卖家收到站内通知。
4. 商品售出或下架后：不再允许新买家提交购买意向；已有意向记录仍可查看当时的联系方式快照，便于交易追溯。

联系方式快照应来自卖家注册时的微信/QQ 字段，并写入意向记录，避免卖家后续修改联系方式导致历史记录失真。前端展示时可以优先展示微信，其次 QQ；若两者都有则同时展示。

## 7. 购买意向与服务费预留

Phase 2 新增轻量购买意向模型，不立即依赖完整 `goods_orders` 支付闭环。建议创建 `goods_intents` 表：

- 买家、商品、卖家；
- 留言；
- 联系方式快照；
- 状态：`OPEN`、`ACCEPTED`、`DECLINED`、`CANCELED`、`COMPLETED`；
- 可选 `service_fee_id`；
- 创建、更新时间。

当买家点击“我想要”时，后端校验商品可交易，创建意向，写入通知。卖家可以在商品详情或“我的发布”中查看意向，选择接受、拒绝或标记商品已售出。

服务费接口预留方式：

- 默认不创建强制服务费。
- 在配置启用后，可以在意向创建或成交确认时创建 `ServiceFeeRecord(target_type = GOODS_INTENT 或 GOODS, target_id = intent.id 或 goods.id)`。
- 支付仍通过现有 `PaymentService` 和外部支付中心边界处理，不在 CampusHub 保存支付宝密钥。

这样可以先上线交易联系能力，同时保留未来收费路径。

## 8. 评价、信用与展示

现有数据库已有 `reviews` 表，但缺少 Java 实体和 API。Phase 2 应补 `Review` 实体、仓库、服务和控制器。

评价规则：

- 买家可以对已完成或卖家标记售出的商品意向进行评价。
- 卖家也可以对买家进行评价，但第一版可以先实现买家评价卖家。
- 每个用户对同一商品/意向只允许评价一次。
- 评价包含 1-5 分和文本内容。
- 商品详情展示卖家信用分、最近评价、平均评分和评价数量。

信用分不应在 Phase 2 中做复杂自动算法。第一版只展示 `users.credit_score`，并计算评价均分/数量作为补充信息。后续可把严重违规、差评和纠纷处理结果纳入信用调整。

## 9. 收藏、评论与举报

### 9.1 收藏

复用 `Favorite` 通用模型，新增写接口：收藏、取消收藏、查询当前用户是否已收藏、收藏数量。商品列表和详情页展示收藏数与收藏状态。

### 9.2 评论

复用 `Comment` 通用模型，新增写接口：发布评论、回复评论、删除自己的评论或管理员隐藏评论。商品详情页展示评论列表，支持一级回复即可，不做复杂楼中楼。

### 9.3 举报

复用 `ReportRecord` 通用模型，新增用户提交举报接口。商品详情页提供举报商品入口，评论项提供举报评论入口，卖家信息区提供举报用户入口。

后台运营页增加二手相关筛选：`target_type = GOODS`、`GOODS_INTENT`、`COMMENT`、`USER`。管理员可以标记举报已处理，并在需要时下架商品或创建违规记录。

## 10. 前端体验设计

### 10.1 商品列表页

现有表格页改为卡片式商品市场：

- 顶部筛选：分类、校区、价格区间、成色、关键词、只看在售。
- 商品卡片：封面图、标题、价格、原价、成色、校区/地点、卖家昵称、信用分、收藏数、浏览量。
- 发布入口：具备发布者身份时显示“发布商品”；否则显示“申请二手发布者身份”。

### 10.2 商品详情页

新增 `/goods/:id`：

- 图片轮播/图库；
- 标题、价格、原价、成色、分类、校区、交易地点、描述；
- 卖家卡片：昵称、信用分、评价均分/数量；
- 联系规则区：未表达意向时显示“提交购买意向后可查看联系方式”；提交后显示微信/QQ 快照；
- 操作区：我想要、收藏、评论、举报；
- 评论区和最近评价区；
- 卖家本人可看到编辑、下架、标记售出、查看意向。

### 10.3 发布页

新增或嵌入发布表单，适合移动端填写：

1. 基础信息：标题、分类、价格、原价、成色；
2. 图片：最多 9 张；
3. 交易信息：校区、交易地点、交易方式；
4. 描述与风险提示；
5. 发布确认：说明联系方式展示规则和平台不托管本金。

## 11. 后端 API 设计

建议新增或扩展：

- `GET /api/goods`：支持筛选和返回卡片摘要。
- `GET /api/goods/{id}`：返回详情，按 viewerId 决定联系方式可见性。
- `POST /api/goods`：发布商品，校验 `GOODS_PUBLISHER`。
- `PUT /api/goods/{id}`：卖家编辑。
- `POST /api/goods/{id}/off-shelf`：卖家下架。
- `POST /api/goods/{id}/mark-sold`：卖家标记售出。
- `POST /api/goods/{id}/intents`：提交购买意向。
- `GET /api/goods/{id}/intents`：卖家查看意向。
- `POST /api/goods/intents/{intentId}/accept|decline|complete|cancel`：意向状态流转。
- `POST /api/files/images`：上传图片。
- `POST /api/files/bindings`：绑定图片到商品。
- `POST /api/interactions/favorites`、`DELETE /api/interactions/favorites`：收藏写接口。
- `POST /api/interactions/comments`：评论写接口。
- `POST /api/reports`：举报提交。
- `POST /api/reviews`、`GET /api/reviews/users/{userId}`：评价写入与展示。

路径可以在实现时结合现有控制器命名微调，但职责边界应保持清晰。

## 12. 测试与验证

后端测试重点：

- 未具备 `GOODS_PUBLISHER` 身份不能发布商品。
- 具备 10 元保证金身份后可以发布商品。
- 商品图片绑定后详情能返回有序图片。
- 买家提交意向后才能看到联系方式快照。
- 已售出/下架商品不能创建新意向。
- 收藏、评论、举报、评价接口能写入并按目标查询。
- 服务费配置关闭时不创建强制费用；配置开启时创建待支付 `ServiceFeeRecord`。

前端验证重点：

- 商品市场卡片和筛选可用。
- 未开通二手发布者身份时发布入口跳转身份页。
- 商品详情在未提交意向前隐藏联系方式，提交意向后展示联系方式。
- 图片、收藏、评论、举报、评价区在桌面和移动端都可用。
- build 通过，仅允许已知 Element Plus/Vite 大 chunk 警告。

部署验证沿用 Phase 1 低影响策略：先本地前端 build，后端若本地无 Maven 则使用服务器 Maven 容器低频测试；部署只重建 backend/web；部署后做 server-local API smoke 和浏览器页面检查。

## 13. 非目标

Phase 2 不做：

- 平台托管二手交易本金；
- 购物车；
- 多商品合并订单；
- 自动物流或配送；
- 强制线上支付货款；
- 复杂信用自动扣分算法；
- 实时聊天；
- 精细推荐算法。

这些能力可作为后续平台扩展，但不应阻塞二手交易可信信息发布与联系闭环上线。

## 14. 自检结论

本设计没有依赖修改已应用的 Flyway 迁移；所有数据库变更应通过新增迁移完成。设计保持 CampusHub 支付边界，不读取或保存支付宝密钥，不引入交易本金托管。Phase 2 聚焦二手业务本身，复用 Phase 1 身份、通知、服务费和治理基础，并保留后续服务费启用路径。
