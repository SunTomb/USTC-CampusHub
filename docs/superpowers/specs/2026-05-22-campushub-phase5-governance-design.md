# CampusHub Phase 5 治理、信用与信任运营设计

日期：2026-05-22

## 1. 目标与边界

Phase 5 的目标是把 CampusHub 已有的举报、违规、信用分、通知和运营后台能力升级为跨业务线可用的治理闭环。Phase 1-4 已经上线跑腿任务、二手交易、学生店铺和项目广告四条业务线，下一阶段的核心风险不再是单个页面能否发布内容，而是平台能否处理真实内测中的骚扰、虚假信息、履约纠纷、恶意评论、违规下架、用户冻结和信用追踪。

本阶段建设统一举报处理队列、举报状态流转、违规记录、信用分调整历史、用户限制状态、管理员操作审计、站内通知闭环、管理端治理工作台和用户侧信用中心。

本阶段不做完整认证授权重构，不做法律级申诉系统，不做机器审核或风控模型，不自动扣除真实保证金，不改变支付边界，不做交易本金托管，也不让 CampusHub 读取或保存支付宝密钥正文。

## 2. 推荐路线

采用“统一治理台 + 信用台账 + 限制状态”的路线，而不是为每条业务线分别做独立举报后台。

- 统一举报：所有业务对象都进入同一队列，使用 `target_type + target_id` 识别对象。
- 状态流转：举报从 `OPEN` 进入 `IN_REVIEW`，最终 `RESOLVED`、`REJECTED` 或 `ESCALATED`。
- 违规闭环：处理属实的举报可以创建违规记录，记录严重程度、处罚动作、信用分变化和保证金影响说明。
- 信用台账：用户当前 `credit_score` 保留在 `users` 表，所有调整写入不可丢失的信用记录表。
- 限制状态：通过独立限制记录表达警告、内容冻结、发布冻结、服务冻结和账号禁用，Phase 5 先用于后台可见和关键发布动作拦截，完整 JWT/RBAC 改造留到 Phase 9。
- 审计闭环：管理员每次处理举报、创建违规、调整信用、变更限制都写入管理员操作审计。
- 通知闭环：举报结果、违规、信用调整、限制变化通过站内通知告知相关用户。

## 3. 用户与角色规则

### 3.1 举报人

注册用户可以举报商品、跑腿任务、学生店铺、服务预约、项目广告、评论和用户。举报创建后默认进入 `OPEN`。举报人可以在用户侧信用中心或“我的举报”区域查看自己提交举报的状态、处理结果和处理说明。举报人不能修改管理员判定，也不能看到被举报人的隐私联系方式或内部审计备注。

### 3.2 被举报人

被举报人可以收到举报处理结果、违规记录、信用分调整和限制变化的站内通知。用户侧信用中心展示当前信用分、近期信用变化、违规记录和当前限制状态。Phase 5 不实现复杂申诉表单，但可以在通知和信用中心提示“如有异议请联系平台管理员”。

### 3.3 管理员

管理员在治理工作台处理跨业务举报，可以认领/开始处理、驳回、标记已解决、升级处理、创建违规、调整信用和设置用户限制。由于当前项目仍处于课程原型到内测平台过渡期，Phase 5 的管理员身份仍沿用现有参数化 adminId / 后台入口模式；严格角色鉴权放到 Phase 9。

## 4. 举报模型

现有 `report_records` 已有举报人、目标类型、目标 ID、原因、描述、状态、处理人和处理时间。Phase 5 使用 V10 新迁移扩展，不修改已应用 V1-V9。

建议变更：

- 将历史 `PENDING` 视为新状态 `OPEN` 的兼容值；迁移中可把现有 `PENDING` 更新为 `OPEN`。
- 新增 `review_note`：管理员处理说明，对用户可见时可作为结果说明。
- 新增 `resolution_type`：处理结果类型，例如 `NO_ACTION`、`CONTENT_REMOVED`、`USER_WARNED`、`USER_RESTRICTED`、`ESCALATED`。
- 新增 `updated_at`：便于队列排序。

状态定义：

- `OPEN`：新举报，等待处理。
- `IN_REVIEW`：管理员已开始处理。
- `RESOLVED`：举报属实或部分属实，已完成处置。
- `REJECTED`：举报不成立或信息不足。
- `ESCALATED`：需要人工线下、辅导员、社团负责人或更高权限继续处理。

后台列表按状态、目标类型、举报原因、创建时间排序和筛选。第一版不做复杂全文检索。

## 5. 违规、信用与限制模型

### 5.1 违规记录

现有 `violation_records` 已有用户、关联举报、违规类型、描述和信用分变化。Phase 5 扩展为可运营台账：

- `severity`：`LOW`、`MEDIUM`、`HIGH`、`CRITICAL`。
- `penalty_type`：`WARNING`、`CONTENT_FREEZE`、`POSTING_FREEZE`、`SERVICE_FREEZE`、`ACCOUNT_DISABLED`、`CREDIT_ONLY`。
- `target_type` / `target_id`：记录违规发生在哪个对象上，避免只依赖 report。
- `admin_id`：创建违规的管理员。
- `deposit_impact_note`：保证金影响说明，仅记录运营说明，不自动扣款。

违规创建后可以同步写信用记录、用户限制记录和站内通知。

### 5.2 信用记录

用户当前信用分仍保存在 `users.credit_score`，默认 100。Phase 5 新增信用调整记录表：

- 用户 ID；
- 关联违规 ID，可为空；
- 调整前分数；
- 调整变化量；
- 调整后分数；
- 原因；
- 操作管理员；
- 创建时间。

信用分第一版限制在 0-100。扣分和加分都必须写入记录，禁止只改 `users.credit_score`。信用记录用于用户信用中心、后台治理台和后续运营分析。

### 5.3 用户限制状态

Phase 5 新增用户限制记录表，而不是只在 `users.status` 写一个不可解释的值。限制类型：

- `WARNING`：警告，不拦截操作。
- `CONTENT_FREEZE`：内容冻结，隐藏或限制部分内容展示，第一版主要后台可见。
- `POSTING_FREEZE`：禁止发布商品、跑腿任务、店铺服务项目、项目广告。
- `SERVICE_FREEZE`：禁止接单、预约服务或作为服务提供者履约。
- `ACCOUNT_DISABLED`：账号禁用，第一版写入状态，完整登录拦截留给 Phase 9 强化。

限制记录包含开始时间、结束时间、原因、关联违规、创建管理员和是否生效。前端用户侧显示当前有效限制。后端在 Phase 5 至少对主要发布/接单/申请入口做轻量拦截，避免明显绕过。

## 6. 管理员操作审计

现有 `SafetyLog` 更偏安全日志。Phase 5 建议新增 `admin_action_logs`，专门记录后台治理动作，避免把业务治理审计混入登录/安全事件。

字段包括：

- 管理员 ID；
- 动作类型，例如 `REPORT_START_REVIEW`、`REPORT_REJECT`、`REPORT_RESOLVE`、`VIOLATION_CREATE`、`CREDIT_ADJUST`、`RESTRICTION_CREATE`；
- 目标类型和目标 ID；
- 操作备注；
- 创建时间。

审计日志只追加，不提供删除。后台治理页可展示最近动作，便于追责和交接。

## 7. 站内通知闭环

Phase 5 继续只做站内通知，不做短信、邮件、浏览器推送或小程序订阅。

通知场景：

- 举报人：举报被受理、被驳回、已处理、已升级。
- 被举报人：举报属实后的违规通知、信用分变化、限制状态变化。
- 管理员可选：严重举报或升级事件进入后台队列即可，不需要站内通知轰炸。

通知内容应简洁说明结果和下一步，不暴露对方联系方式、隐私或内部审计信息。

## 8. 管理端治理工作台

推荐新增 `/admin/governance` 页面，而不是继续把所有内容塞进 `/admin/ops`。`/admin/ops` 保留业务监控与指标，`/admin/governance` 聚焦处理队列和动作闭环。

页面结构：

1. 顶部指标卡：开放举报数、处理中举报数、今日处理数、高严重违规数、当前受限用户数。
2. 举报队列：状态、目标类型、举报人、原因、创建时间、处理人、操作按钮。
3. 举报详情抽屉：举报描述、目标信息摘要、历史处理记录、处理说明。
4. 处理动作：开始处理、驳回、解决、升级、创建违规。
5. 违规与信用动作：选择违规类型、严重程度、处罚动作、信用分变化、限制类型、限制时长、通知文案。
6. 审计日志：展示最近管理员动作。

第一版不强求在详情里预览所有目标对象完整内容；可先展示 target type/id 和跳转入口，避免跨业务聚合过重。

## 9. 用户侧信用中心

新增 `/credit` 或 `/trust` 页面，推荐 `/credit`，导航命名为“信用中心”。

内容包括：

- 当前信用分和状态说明；
- 当前有效限制；
- 信用调整记录；
- 违规记录；
- 我提交的举报及状态；
- 平台安全提示：交易本金线下自担、不要泄露验证码、优先保留平台内关键状态记录。

信用中心不展示他人隐私，不提供复杂申诉表单。后续 Beta 阶段可扩展成“申诉/客服工单”。

## 10. 后端 API 设计

建议新增或扩展：

- `GET /api/admin/governance/reports`：统一举报队列，支持状态和目标类型筛选。
- `GET /api/admin/governance/reports/{id}`：举报详情。
- `POST /api/admin/governance/reports/{id}/start-review?adminId=`：进入处理。
- `POST /api/admin/governance/reports/{id}/reject?adminId=`：驳回。
- `POST /api/admin/governance/reports/{id}/resolve?adminId=`：解决，可附带处理类型和说明。
- `POST /api/admin/governance/reports/{id}/escalate?adminId=`：升级。
- `POST /api/admin/governance/violations?adminId=`：创建违规，可选同步信用和限制。
- `POST /api/admin/governance/users/{userId}/credit-adjustments?adminId=`：单独信用调整。
- `POST /api/admin/governance/users/{userId}/restrictions?adminId=`：创建用户限制。
- `GET /api/admin/governance/audit-logs`：管理员动作日志。
- `GET /api/credit/users/{userId}`：用户信用中心汇总。
- `GET /api/credit/users/{userId}/reports`：用户提交的举报状态。

路径可结合现有 `ModerationController` 和 `OperationsController` 微调，但管理端治理动作应集中到一个服务，避免散落到各业务控制器。

## 11. 与现有业务线的关系

Phase 5 不重写四条业务线，而是提供统一治理能力：

- 跑腿任务：任务、任务异常、接单者和发布者都可进入举报/限制/信用闭环。
- 二手交易：商品、卖家、评论、交易意向相关纠纷进入同一队列。
- 学生店铺：店铺、服务项目、预约订单、商家和客户纳入统一处理。
- 项目广告：广告、评论和发布者复用 Phase 5 治理。

主要业务接口只需要在关键动作前查询用户有效限制，例如发布、接单、申请、创建店铺服务、提交项目广告。不要在 Phase 5 做全量鉴权改造。

## 12. 测试与验证

后端测试重点：

- 新举报默认进入 `OPEN`。
- 管理员可以把举报流转为 `IN_REVIEW`、`REJECTED`、`RESOLVED`、`ESCALATED`。
- 解决举报时创建违规记录，并同步写信用记录、用户限制和管理员审计。
- 信用分调整保持 0-100 边界，并记录调整前后分数。
- 用户信用中心返回当前分数、限制、违规、信用记录和举报摘要。
- 站内通知在举报处理、违规、信用和限制变化时创建。

前端验证重点：

- `/admin/governance` 能查看举报队列、筛选、处理、创建违规和查看审计。
- `/credit` 能展示信用分、限制、违规、信用记录和我的举报。
- 现有 `/goods`、`/tasks`、`/shops`、`/project-ads`、`/admin/ops` 仍能渲染。
- 移动端 `/credit` 无明显横向溢出。

生产验证仍按小服务器低影响策略：先本地/代码审查，必要时推送后做服务器 Docker build、服务端 API smoke 和 Playwriter 浏览器验证；不要本地安装依赖，不读取任何 secret。

## 13. 非目标

Phase 5 不做：

- 全量 JWT/RBAC 改造；
- 管理员权限矩阵；
- 自动内容审核或机器学习风控；
- 法律级申诉、仲裁和证据链系统；
- 自动扣除保证金或真实资金处罚；
- 支付中心接口改造；
- 交易本金托管；
- 短信、邮件、微信通知；
- 小程序客户端；
- 复杂数据分析导出。

这些能力可以在 Phase 6-10 中分别处理，不能混入 Phase 5 扩大范围。

## 14. 自检结论

本设计聚焦单一 Phase 5 子系统，范围与 Phase 4 接近；复用现有 `moderation`、`notification`、`user.credit_score` 和 target model；数据库变更通过 V10 新迁移完成，不修改 V1-V9；保留 CampusHub 不托管本金、不按单冻结押金、不读取支付宝密钥正文的边界；认证授权硬化留给 Phase 9，避免本阶段破坏现有演示和内测流程。
