# CampusHub Phase 13 身份感知 UI / UX 终极优化设计

日期：2026-05-24

## 1. 背景与目标

Phase 1-12 已经让 CampusHub 具备真实校园服务平台的核心闭环：公开浏览、邮箱注册登录、跑腿任务、二手交易、学生店铺、项目广告、站内通知、信用治理、运营后台、钱包充值提现、支付宝支付中心跳转和微信人工充值。当前主要问题不再是业务能力缺失，而是产品观感、身份分层和导航呈现仍带有课程 demo 痕迹。

Phase 13 的目标是进行一次以视觉品牌为优先的 UI / UX 终极优化：让 CampusHub 看起来像可运营的高级校园服务平台，让游客、普通学生、跑腿接单者、二手发布者、店铺商家和管理员进入系统后能清楚理解自己的身份、能力、下一步动作和可访问入口。

本阶段不追求新增业务域，而是把现有业务能力重新包装为更可信、更高级、更顺畅的产品体验。重点是统一品牌视觉、统一卡片与按钮层级、统一空状态和说明文案、按身份过滤导航、优化无权限引导，以及打磨注册/登录、充值、身份申请、发布/管理、通知和管理员审核等核心路径。

## 2. 设计原则

1. 视觉品牌优先。优先解决“像课程 demo”的观感问题：品牌壳层、首页、卡片、列表、表单、后台页面要形成统一气质，而不是 Element Plus 控件堆叠。
2. 复用现有身份，不重建权限系统。前端以 `/api/auth/me` 返回的 `roles` 为身份事实来源，不新增数据库表或新的角色模型。
3. 一个导航配置驱动多端。桌面侧栏、移动底部 tab、移动“更多”目录、首页快捷入口和权限提示应尽量复用同一份前端导航/能力配置，避免入口不一致。
4. 隐藏不相关入口，解释不可用能力。游客和普通学生不应该在主导航看到所有后台和工作台入口；用户主动进入无权限页面时，应看到登录、申请身份、支付保证金或等待审核的清晰引导。
5. 移动端像真实产品，桌面端像运营工作台。移动端强调高频路径和底部 tab；桌面端强调信息密度、运营效率和高级管理面板。
6. 保持生产资金和安全边界。支付宝仍由 API-Transfer-Station 支付中心处理；CampusHub 不读取、不保存支付宝密钥，不打印或提交真实 `.env`、SMTP 密码、JWT secret、payment-center token/signing secret、数据库密码或二维码敏感内容。

## 3. 现状判断

当前前端已经具备 Phase 7 的响应式基础和共享组件：`EmptyState`、`FormSection`、`PageActions`、移动底部 tab、移动目录抽屉和集中式 `styles.css`。认证状态由 `frontend/src/stores/auth.ts` 管理，当前用户角色通过 `auth.currentUser.roles` 暴露。

主要缺口：

1. `MainLayout.vue` 中导航项是静态数组，所有用户几乎看到同一套入口。
2. 路由守卫只区分 `requiresAuth` 和 `requiresAdmin`，缺少角色能力级提示。
3. 首页仍是简单模块卡片，没有承担真实产品 landing、身份工作台和转化引导职责。
4. 角色申请、钱包、发布/管理、管理员页面虽然功能齐全，但视觉层级和运营文案不够统一。
5. 无权限路径主要通过消息提示和跳转处理，缺少可理解的页面级引导。
6. 移动端底部 tab 固定展示通用入口，未体现游客、学生、业务身份和管理员差异。

## 4. 身份与能力模型

Phase 13 不新增后端角色，只在前端建立身份显示与能力映射。

### 4.1 身份类型

1. 游客：未登录用户。可浏览首页、政策说明、二手、跑腿、店铺、项目广告公开列表和详情，可进入登录注册。
2. 普通学生：已登录但没有业务角色的用户。可使用钱包、通知、信用中心、角色申请、收藏/评论/举报等基础功能。
3. 跑腿接单者：拥有 `ROLE_RUNNER` 或后端实际返回的跑腿角色标识。可看到跑腿大厅、任务工作台、接单/申请相关入口。
4. 二手发布者：拥有 `ROLE_GOODS_PUBLISHER` 或后端实际返回的二手发布角色标识。可看到商品发布和二手管理相关入口。
5. 店铺商家：拥有 `ROLE_SHOP_MERCHANT` 或后端实际返回的店铺商家角色标识。可看到店铺工作台、预约处理和服务项目维护入口。
6. 管理员：拥有 `ROLE_ADMIN`。可看到审核、运营、治理、支付监控、钱包运营等管理入口。

用户可能同时拥有多个身份。UI 应展示“当前已解锁能力”而不是强迫用户切换单一身份。后续如需明确当前身份切换，可作为未来 Phase 独立设计。

### 4.2 能力卡片

在首页、侧栏用户区域和角色申请页展示统一的能力状态：

- 已解锁：显示身份徽章、可进入的工作台和下一步建议。
- 可申请：显示保证金金额、自动开通或人工审核说明、申请入口。
- 待处理：如果已有申请或支付单信息可从现有页面获得，则在角色申请页展示待支付/待审核状态；全局壳层不强依赖额外 API。
- 不可访问：显示原因和解决路径，例如“需要先登录”“需要开通二手发布者身份”“店铺商家需管理员审核”。

## 5. 导航与信息架构

### 5.1 统一导航配置

新增或整理前端导航配置，字段建议包括：

- `path`：路由路径；
- `label`：显示名称；
- `group`：公开浏览、我的校园、业务工作台、管理后台；
- `audience`：guest、student、runner、goodsPublisher、shopMerchant、admin；
- `mobilePriority`：是否进入移动底部 tab；
- `requiredAuth`、`requiredRole`、`requiredAdmin`：权限要求；
- `lockedTitle`、`lockedDescription`、`primaryAction`：无权限引导文案。

该配置驱动桌面侧栏、移动 tab、移动目录抽屉、首页快捷入口和权限页面，减少未来维护成本。

### 5.2 游客导航

游客只看到：

- 首页；
- 二手商品；
- 悬赏任务；
- 学生店铺；
- 项目广告；
- 协议与风险；
- 登录注册。

游客移动端底部 tab 建议为：首页、跑腿、二手、店铺、登录、更多。通知、钱包、信用、角色、后台不在主导航展示。

### 5.3 登录学生导航

普通学生看到公开浏览入口和“我的校园”入口：

- 钱包；
- 通知；
- 信用；
- 身份保证金；
- 协议与风险。

移动端底部 tab 建议为：首页、跑腿、二手、店铺、通知、更多；更多目录中展示钱包、信用、角色申请和公开业务。

### 5.4 业务身份导航

拥有对应身份后展示对应工作台：

- 跑腿接单者：跑腿大厅、我的接单/任务工作台；
- 二手发布者：二手商品、发布商品、我的二手管理入口；
- 店铺商家：学生店铺、商家工作台、预约处理；
- 项目广告发布暂按登录学生能力处理，若后端后续新增独立项目发布身份，再纳入同一配置。

业务入口应以“工作台”方式呈现，而不是混在普通浏览入口里。桌面侧栏按分组显示；移动端更多目录按分组显示。

### 5.5 管理员导航

管理员额外看到“管理后台”分组：

- 内容审核；
- 运营数据；
- 治理台；
- 支付监控；
- 钱包运营。

管理员移动端底部 tab 不应被后台入口挤满，仍保留高频用户侧入口；管理后台放在“更多”目录的高优先分组中。桌面端则突出后台矩阵和运营效率。

## 6. 视觉系统

### 6.1 品牌方向

建议采用“清爽校园科技 + 可信运营平台”方向：

- 主色：深蓝 / 靛蓝用于信任和运营后台；
- 辅色：青绿用于校园服务和交易完成感；
- 点缀：暖橙用于保证金、充值、待处理和 CTA；
- 背景：浅灰蓝内容画布，局部使用柔和渐变；
- 字体：继续使用 Inter / PingFang SC / Microsoft YaHei，强化标题字重和层级。

### 6.2 全局壳层

桌面端：

- 左侧侧栏升级为品牌导航区，包含平台标识、简短 slogan、当前用户身份徽章和分组导航。
- 顶部 header 从单一渐变条升级为产品状态栏：页面上下文、当前身份摘要、通知/钱包/登录操作。
- 内容区采用最大宽度和分层 surface，避免大片空白或控件直贴边。

移动端：

- 顶部展示品牌、当前身份和登录状态；
- 底部 tab 按身份生成；
- “更多”抽屉使用分组目录和身份能力卡，而不是简单菜单列表。

### 6.3 卡片体系

统一三类卡片：

1. 市场卡片：用于二手、跑腿、店铺、项目广告公开列表，突出标题、金额/价格、校区、信用、状态和主操作。
2. 工作台卡片：用于身份工作台、管理后台、钱包和通知，突出待办、状态、下一步动作。
3. 说明/空状态卡：用于未登录、无权限、空列表、风险提示和政策说明，提供明确 CTA。

卡片应减少默认 `el-card` 的边框感，通过统一圆角、阴影、渐变头部、标签和按钮层级形成品牌一致性。

### 6.4 按钮与状态层级

按钮层级：

- 主 CTA：注册登录、申请身份、充值、发布、进入工作台、管理员处理；
- 次 CTA：查看详情、继续支付、筛选、重试；
- 风险动作：下架、拒绝、限制、驳回等使用清晰但克制的危险色。

状态标签统一中文表达：待支付、已解锁、待审核、可接单、处理中、已完成、需管理员处理等。避免直接暴露过多英文枚举，除非用于支付 provider 等运营诊断信息。

## 7. 核心页面设计

### 7.1 首页

首页从静态模块列表升级为身份感知 landing / workbench。

游客首页：

- 顶部 hero：说明 CampusHub 是校园二手交易与学生微服务平台；
- 主 CTA：注册 / 登录；
- 次 CTA：浏览二手、找跑腿、看店铺、看项目；
- 信任说明：校内邮箱、信用治理、站内通知、钱包边界、交易风险提示。

登录学生首页：

- 欢迎区展示昵称、校园邮箱、已解锁身份；
- 下一步建议：充值钱包、申请身份、查看通知、完善信用；
- 业务快捷入口：发布跑腿、发布二手、创建店铺、查看项目广告。

管理员首页：

- 增加管理入口矩阵：审核、治理、运营、支付、钱包；
- 提示低影响运营原则和风险边界；
- 不在首页重复加载大量后台数据，避免额外 API 压力。

### 7.2 登录注册

`/auth` 继续承载登录和注册，但视觉升级为品牌 onboarding：

- 左侧或顶部展示“为什么加入 CampusHub”；
- 注册路径强调校园邮箱验证码、微信/QQ 至少一项联系信息、隐私边界；
- 登录支持用户名或校园邮箱的文案明确化；
- 登录成功后引导到首页工作台或原目标页面。

### 7.3 钱包与充值

`/wallet` 视觉升级为资金中心：

- 顶部展示可用余额、冻结余额、累计充值/提现等现有可得信息；
- 支付宝充值入口保留 `PAYMENT_CENTER` 和继续支付能力；
- 微信充值入口保留二维码和“备注校园邮箱或 CampusHub 用户名”的说明；
- 风险文案继续强调 CampusHub 不直接处理支付宝密钥，微信充值需人工审核。

### 7.4 身份申请

`/roles` 升级为“能力解锁中心”：

- 顶部展示当前已解锁身份；
- 三张身份卡分别展示可做什么、保证金金额、自动开通/人工审核规则、适合谁；
- 已拥有身份显示“已解锁”而不是继续申请；
- 未拥有身份显示“申请开通”或“支付保证金”；
- 支付返回 payUrl 时继续沿用新窗口打开行为。

### 7.5 跑腿、二手、店铺、项目广告

公开列表继续允许游客浏览，但页面 CTA 按身份变化：

- 游客：主要 CTA 是登录后发布/接单/预约/收藏；
- 普通学生：提示申请对应身份或进入基础互动；
- 已解锁身份：显示发布、接单、工作台、管理入口；
- 管理员：不干扰用户侧浏览，但可在更多目录进入管理后台。

发布和管理页应使用统一表单分区、风险说明和身份提示。无对应身份时，页面展示申请入口而不是只弹错误。

### 7.6 通知与信用

`/notifications` 和 `/credit` 升级为“我的校园状态”：

- 空状态说明为什么没有通知/记录；
- 通知列表强调待处理、系统提醒、支付/身份/审核结果；
- 信用中心强调信用分、限制状态、违规记录和申诉/报告路径。

### 7.7 管理后台

管理页面不追求移动端复杂操作极致效率，但必须有高级运营台观感：

- `/admin/ops`：运营数据使用更统一的指标卡和分区；
- `/admin/governance`：治理队列突出风险等级、处理状态和管理员动作；
- `/admin/review`：审核入口明确；
- `/admin/payment`：支付订单和回调事件保持运营诊断感；
- `/admin/wallet`：充值、提现、冻结记录使用清晰状态标签。

桌面端保留表格效率；移动端使用内部横向滚动和分区卡片避免页面级横向溢出。

## 8. 路由权限与引导

Phase 13 扩展前端路由 meta，但不改变后端安全边界。

建议 meta 字段：

- `requiresAuth`：需要登录；
- `requiresAdmin`：需要管理员；
- `requiredRole`：需要业务身份；
- `lockedTitle` / `lockedDescription`：无权限说明；
- `unlockRoute`：推荐跳转，例如 `/roles`；
- `returnTo`：登录后回到原目标。

守卫行为：

1. 未登录访问私有页：跳转 `/auth`，保留目标路径，显示登录后继续的提示。
2. 非管理员访问管理页：进入统一无权限引导或回首页并展示明确说明，不出现白屏。
3. 缺少业务身份访问工作台：展示角色申请引导，说明保证金和审核规则。
4. API 返回 401/403：继续由 Axios/auth store 清理或提示，但页面层要尽量有可恢复入口。

## 9. 技术方案

主要修改前端，不新增 Flyway 迁移。

建议新增或整理：

- `frontend/src/config/navigation.ts`：身份感知导航与能力配置；
- `frontend/src/utils/identity.ts` 或在 auth store 中增加 getter：身份判断、角色标签、能力摘要；
- `frontend/src/components/common/IdentityBadge.vue`：身份徽章；
- `frontend/src/components/common/LockedState.vue`：无权限/需申请身份引导；
- `frontend/src/components/common/CapabilityCard.vue`：首页和角色页复用的能力卡；
- 继续复用 `EmptyState`、`FormSection`、`PageActions`。

重点修改：

- `frontend/src/layouts/MainLayout.vue`：身份区、分组导航、移动 tab 和更多目录；
- `frontend/src/router/index.ts`：扩展 meta 和守卫提示；
- `frontend/src/stores/auth.ts`：增加角色/身份 getter；
- `frontend/src/styles.css`：品牌视觉、壳层、卡片、按钮、状态、后台面板；
- `frontend/src/views/HomeView.vue`：身份感知首页；
- `frontend/src/views/AuthView.vue`、`WalletView.vue`、`RoleApplicationsView.vue`、业务列表/详情/工作台、管理员页面：按优先级套用视觉和身份引导。

后端原则：只有在现有 `CurrentUser.roles` 无法区分需要的身份时才做最小接口调整。当前判断前端信息基本足够，因此 Phase 13 默认不改后端、不新增 V13 迁移。

## 10. 实施顺序

1. 建立身份工具和导航配置，确认现有角色枚举名称并兼容后端实际返回值。
2. 升级全局壳层：桌面侧栏、顶部身份区、移动 tab、更多目录和分组导航。
3. 升级全局视觉基础：颜色、surface、卡片、按钮、标签、空状态、后台面板和移动规则。
4. 重做首页为身份感知 landing / workbench。
5. 打磨登录注册、钱包充值和身份申请三条新用户关键路径。
6. 打磨跑腿、二手、店铺、项目广告的入口 CTA、无权限引导和发布/工作台视觉。
7. 打磨通知、信用和管理员后台页面的统一观感。
8. 跑前端构建；部署后进行 server-local API smoke 和 Playwriter 桌面/移动验证。
9. 更新 README、CLAUDE handoff，并提交 Phase 13 结果。

## 11. 验证计划

本地可用时运行：

- `npm --prefix frontend run build`。

如本地后端 Maven 不可用，不强行安装。完整生产验证遵守低影响原则：本地修改并提交 GitHub，服务器 `/opt/campushub` 拉取，targeted Docker build/restart，server-local API smoke，Playwriter 检查。

API smoke 至少覆盖：

- `/api/goods`；
- `/api/tasks`；
- `/api/shops`；
- `/api/project-ads`；
- `/api/payment/service-fees`；
- `/api/admin/ops/analytics/overview`；
- `/api/admin/governance/dashboard`；
- `/api/credit/users/1`。

## 12. 非目标

Phase 13 不做：

- 新业务线；
- 微信小程序或原生 App；
- 替换 Element Plus；
- 企业级 RBAC 重构；
- 后端权限模型重写；
- 新数据库迁移，除非实现中发现不可避免的后端身份信息缺口；
- CampusHub 直连支付宝密钥；
- 打印、读取、复制或提交真实生产 secrets；
- 压力测试或高频生产探测；
- 实时聊天、地图 POI、复杂推荐算法。

## 13. 自检

本设计聚焦 Phase 13 的 UI / UX 终极优化，优先级明确为视觉品牌升级，并用身份感知导航与权限引导支撑真实运营感。方案主要改前端，不改变生产支付边界，不编辑已应用 V1-V12 迁移，不依赖读取 secrets。身份模型复用现有 `roles`，避免后端大改。验证计划覆盖游客、普通学生、管理员、移动端无横向溢出和支付宝/微信充值回归，符合当前生产约束与 Phase 13 目标。
