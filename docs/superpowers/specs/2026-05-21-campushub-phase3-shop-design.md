# CampusHub Phase 3 学生店铺升级设计

日期：2026-05-21

## 1. 目标与边界

Phase 3 的目标是把学生店铺从静态展示升级为可内测的校园技能服务预约平台。平台继续保持 CampusHub 总边界：不托管服务本金，不做平台内全额结算，不做逐单保证金冻结。CampusHub 负责商家身份准入、店铺信息结构化、服务项目展示、预约流程、联系方式门槛、站内通知、评价举报、运营后台和服务费接口预留。

本阶段聚焦摄影、贴膜、电脑维护、妆造、家教、设计、剪辑等持续服务。用户通过店铺页浏览服务项目，提交预约意向；商家在工作台确认、拒绝、完成或取消预约；双方后续通过微信/QQ 或线下沟通完成服务本金结算。平台记录关键行为并为后续服务费、治理和信用沉淀留接口。

## 2. 推荐路线

采用“商家准入 + 服务橱窗 + 预约工作台”的路线，而不是直接建设复杂排班、在线支付和实时聊天。

- 商家准入：只有 `SHOP_MERCHANT` 身份已支付 20 元保证金且人工审核通过的用户可以创建/管理店铺。
- 店铺主流程：商家开店、维护店铺主页、发布服务项目、设置服务范围与联系方式展示规则；用户浏览、收藏/评论、提交预约；商家处理预约；完成后双方可评价和举报。
- 服务费边界：Phase 3 只预留店铺服务费记录和支付中心接口，不强制托管服务本金。若后续启用平台服务费，可在预约确认或完成节点创建 `ServiceFeeRecord(target_type = SERVICE_ORDER, target_id = order.id)`。
- 治理闭环：店铺、服务项目、预约和用户都可被举报；后台可查看商家审核、店铺状态、预约异常、评价与举报。

## 3. 用户与权限规则

### 3.1 店铺商家身份

用户必须先申请 `SHOP_MERCHANT` 身份，支付 20 元身份保证金后进入人工审核。后端创建店铺、发布服务项目、处理预约时必须校验 `role_applications` 中该用户 `role_type = SHOP_MERCHANT`、`deposit_status = PAID`、`review_status = APPROVED`。

前端点击“开设店铺”时，如果未通过商家身份，应显示引导卡片并跳转身份保证金页；后端仍是最终权限边界。

### 3.2 顾客身份

普通注册用户即可浏览店铺、查看服务项目、提交预约意向、评价和举报。顾客不需要额外保证金。

### 3.3 管理员能力

管理员可以查看待审核商家、店铺列表、预约异常、举报、评价和违规记录；可以冻结违规店铺、处理举报、记录违规。复杂自动风控放到后续阶段。

## 4. 店铺与服务模型

现有 `shops`、`service_items`、`service_orders` 已有基础表和实体。Phase 3 应通过新 Flyway 迁移扩展，不能修改已应用 V1-V6。

建议新增或强化字段：

- `shops.campus_zone`：主要服务校区。
- `shops.contact_visibility`：联系方式展示规则，默认 `ORDER_ONLY`。
- `shops.cover_file_id`：店铺封面，复用文件资源。
- `shops.opening_hours`：服务时间文字说明。
- `shops.contact_snapshot` 或按订单生成联系方式快照。
- `service_items.category`：服务类型，如摄影、贴膜、维修、妆造、家教等。
- `service_items.price_unit`：计价单位，如 次、小时、套、面议。
- `service_items.min_price` / `max_price`：价格区间，允许面议。
- `service_items.cover_file_id`：项目封面。
- `service_orders.contact_snapshot`：预约创建时的商家联系方式快照。
- `service_orders.cancel_reason`：取消原因。
- `service_orders.service_fee_id`：服务费记录预留。

店铺状态建议：

- `DRAFT`：草稿。
- `PENDING_REVIEW`：待审核。
- `APPROVED`：营业中。
- `PAUSED`：商家暂停接单。
- `BLOCKED`：平台冻结。
- `CLOSED`：关闭。

服务项目状态建议：`DRAFT`、`PUBLISHED`、`PAUSED`、`OFF_SHELF`、`BLOCKED`。

预约状态建议：`REQUESTED`、`ACCEPTED`、`REJECTED`、`IN_SERVICE`、`COMPLETED`、`CANCELED`、`DISPUTE_HANDLING`。

## 5. 联系方式展示规则

联系方式不应在店铺列表公开展示。推荐规则：

1. 商家本人查看自己的店铺/预约：可看到完整联系方式。
2. 普通用户未预约：只能看到联系规则提示、评论、评价和举报入口。
3. 用户提交预约后：系统创建预约记录和联系方式快照，顾客可看到商家微信/QQ 快照，商家可看到顾客联系方式摘要。
4. 预约取消或完成后：历史预约继续保留当时联系方式快照，便于追溯；店铺页不公开展示。

## 6. 预约流程

第一版预约流程不做复杂档期系统，采用用户选择期望时间 + 商家确认的轻量模式。

1. 顾客在服务项目页选择期望时间、补充说明并提交预约。
2. 系统创建 `REQUESTED` 预约，给商家站内通知。
3. 商家在工作台接受或拒绝。接受后预约进入 `ACCEPTED`，顾客收到通知。
4. 服务进行时可进入 `IN_SERVICE`，完成后由商家或顾客发起完成，另一方确认或直接标记完成。
5. 取消、异常和纠纷进入后台可查队列。

后续可在真实数据积累后再做档期表、可预约时间段、自动冲突检测和日历视图。

## 7. 前端体验设计

### 7.1 店铺列表页

现有 `/shops` 从简单列表升级为店铺市场：

- 顶部筛选：服务类型、校区、关键词、评分、是否营业。
- 店铺卡片：封面、店铺名、服务范围、校区、评分、商家昵称、服务项目数、近期评价。
- 开店入口：未通过商家身份显示“申请店铺商家身份”；已通过显示“管理我的店铺”。

### 7.2 店铺详情页

新增或完善 `/shops/:id`：

- 店铺头图、名称、描述、服务范围、校区、营业时间、评分。
- 服务项目橱窗：标题、价格区间、时长、分类、封面、预约按钮。
- 联系规则提示：预约后可查看联系方式。
- 评论、评价、举报入口。
- 商家本人可进入编辑店铺、管理服务项目和预约工作台。

### 7.3 商家工作台

新增 `/shops/merchant` 或 `/shops/:id/manage`：

- 店铺资料维护。
- 服务项目创建/编辑/上下架。
- 预约列表与状态流转。
- 异常/取消处理。
- 基础经营数据：预约量、完成量、评价均分、待处理预约。

### 7.4 移动端

用户侧页面优先适配手机浏览器：店铺卡片、服务项目、预约弹窗和通知入口必须无横向溢出。后台/商家工作台桌面优先，但关键预约处理按钮在手机端也应可用。

## 8. 后端 API 设计

建议新增或扩展：

- `GET /api/shops`：支持筛选并返回店铺卡片摘要。
- `POST /api/shops`：创建店铺，校验 `SHOP_MERCHANT`。
- `GET /api/shops/{id}`：店铺详情。
- `PUT /api/shops/{id}`：商家编辑店铺。
- `POST /api/shops/{id}/pause|resume|close`：店铺状态流转。
- `POST /api/shops/{id}/service-items`：创建服务项目。
- `PUT /api/service-items/{id}`：编辑服务项目。
- `POST /api/service-items/{id}/publish|pause|off-shelf`：项目状态流转。
- `POST /api/service-items/{id}/orders`：顾客提交预约。
- `GET /api/shops/{id}/orders`：商家查看预约。
- `POST /api/service-orders/{id}/accept|reject|start|complete|cancel`：预约状态流转。
- `POST /api/moderation/reports`：复用举报接口。
- `POST /api/reviews`、`GET /api/reviews/users/{userId}`：复用评价接口。
- `GET /api/admin/ops/shop-orders`：后台预约监控。

路径可在实现时结合现有控制器微调，但职责边界应清晰。

## 9. 测试与验证

后端测试重点：

- 未通过 `SHOP_MERCHANT` 不能创建店铺或发布服务项目。
- 20 元保证金且人工审核通过后可以创建店铺。
- 顾客提交预约后才能看到联系方式快照。
- 非商家本人不能处理该店铺预约。
- 已冻结/下架的店铺或服务项目不能创建新预约。
- 接受、拒绝、完成、取消预约会写入状态并创建站内通知。
- 服务费配置关闭时不创建强制费用；启用时在指定节点创建待支付服务费。

前端验证重点：

- 店铺市场筛选和卡片可用。
- 未开通商家身份时开店入口跳转身份页。
- 店铺详情和服务项目预约流程可用。
- 商家工作台能处理预约。
- 桌面和移动端无明显横向溢出。
- `npm --prefix frontend run build` 通过，仅允许既有大 chunk 警告。

## 10. 非目标

Phase 3 不做：

- 服务本金托管；
- 每笔预约保证金冻结；
- 复杂排班/日历冲突系统；
- 平台内实时聊天；
- 自动派单或推荐；
- 多校区扩张；
- 小程序客户端。

这些能力可在真实预约数据积累后逐步补充，不应阻塞学生店铺内测闭环上线。

## 11. 自检结论

本设计保持 CampusHub 支付边界，不读取或保存支付宝密钥，不引入交易本金托管；数据库变更必须通过 V7+ 新迁移完成；店铺商家身份沿用 Phase 1 的 20 元保证金 + 人工审核规则；位置仍采用校区枚举和文字说明优先；通知仍采用站内通知优先。
