# CampusHub Phase 7 响应式 Web 与用户体验设计

日期：2026-05-22

## 1. 背景与目标

Phase 1-6 已经让 CampusHub 具备跑腿、二手、学生店铺、项目广告、治理信用、运营分析六个主要能力面。Phase 7 不继续增加业务闭环，而是把现有 Web 应用打磨到适合真实校园内测的移动浏览器体验。

本阶段目标是：学生在手机上能顺畅完成注册登录、浏览、发布、接单/预约/联系、查看通知与信用；管理员在桌面端继续保持运营效率；所有关键页面在移动端不出现明显横向溢出、难以点击、表单过长无层次、空白无解释、错误信息不一致等问题。

## 2. 设计原则

1. 移动优先，不牺牲桌面管理效率。用户侧页面优先适配 390x844 手机视口，后台页面在移动端保证可读可操作，桌面端继续保留表格效率。
2. 小步复用，不做全量设计系统重写。优先抽取共享 CSS 类和少量轻量组件，避免重写 Element Plus 或改变整体技术栈。
3. 只打磨现有流程，不增加新业务域。Phase 7 不做支付中心强化、JWT/RBAC 改造、微信小程序、地图 POI、复杂推荐或新数据库表。
4. 明确状态和下一步。加载、空状态、未登录、无权限、表单校验、提交失败都要给出可理解的提示和操作入口。
5. 保持资金与隐私边界。所有文案继续强调 CampusHub 不托管交易本金、不做逐单保证金冻结，联系方式仅在达成业务条件后展示。

## 3. 范围

### 3.1 重点页面

用户侧：`/auth`、`/tasks`、`/tasks/:id/workspace`、`/goods`、`/goods/:id`、`/goods/publish`、`/shops`、`/shops/:id`、`/shops/merchant`、`/project-ads`、`/project-ads/:id`、`/project-ads/manage`、`/wallet`、`/roles`、`/notifications`、`/credit`。

管理侧：`/admin/ops`、`/admin/governance`、`/admin/review`。

### 3.2 主要体验问题

当前全局样式已经有基础响应式规则，但仍存在几个 Phase 7 应优先处理的问题：移动端完整侧边菜单占用首屏；卡片、列表、表单和操作区行为不完全一致；表格和筛选栏在窄屏下需要更好折行或内部滚动；发布类表单字段多且层次不够；空列表、加载失败、未登录提示文案不统一；对话框、按钮标签和低对比辅助文字需要基础可访问性检查。

## 4. 推荐方案

采用“共享响应式基础 + 关键页面逐步套用”的方案。

1. 全局布局：移动端将侧边导航压缩为顶部菜单入口，减少首屏占用；页面主内容统一安全宽度、间距和滚动策略。
2. 共享样式：新增或整理状态卡片、操作区、表单分区、移动表格包装等类。
3. 用户侧核心流程：优先改 `/auth`、`/tasks`、任务工作台、`/goods`/详情/发布、`/shops`/详情/商家、`/project-ads`/详情/管理。
4. 账户与运营辅助页：随后改 `/wallet`、`/roles`、`/notifications`、`/credit`、后台运营和治理页面。
5. 验证矩阵：桌面、平板-ish、390x844 手机逐页检查，重点观察横向溢出、按钮可点击、表单可提交、弹窗可关闭、空状态可理解。

## 5. 页面设计细节

### 5.1 全局布局与导航

桌面端继续使用左侧导航和顶部头部。移动端不再让完整侧边栏占据首屏，而是将品牌、登录状态和菜单入口压缩到顶部区域。菜单使用 Element Plus 抽屉承载，导航项保持现有路由顺序。

页面内容区在移动端统一减少 padding，避免卡片边距叠加导致内容过窄。标题区采用纵向布局：上方是页面标题和说明，下方是主要操作按钮。主操作按钮在手机端尽量全宽或至少有足够点击面积。

### 5.2 卡片、列表和筛选

商品、任务、店铺、项目广告继续以卡片为主。移动端统一一列卡片，卡片内优先展示决策信息：标题、金额/报酬/价格、校区、状态、信用/浏览/联系条件、主操作。次要统计信息折到第二行或小号标签，不强行塞入单行。

筛选栏在桌面端保持横向布局，移动端改为一列或两列紧凑布局，关键词输入优先占满宽度。筛选按钮和重置按钮靠近筛选控件，避免用户滚动后找不到提交入口。

### 5.3 表单发布流程

跑腿、二手商品、店铺服务项目、项目广告发布/管理表单采用统一分区：基本信息；校区/地点/范围；金额、服务费或价格提示；联系方式展示与风险提示；附件/备注；提交确认。

不在 Phase 7 引入多步向导，以免扩大范围；但通过分区标题、帮助文案、必填提示和底部操作条，让长表单在手机端更容易完成。表单提交失败时优先显示后端错误文案；前端补充一致的必填提示。

### 5.4 状态反馈

新增统一的空状态和登录提示模式：空列表说明当前为什么没有数据，并给出下一步；未登录时提示登录后可执行哪些操作，并提供跳转 `/auth`；加载失败保留错误详情，并提供重试按钮；提交中按钮使用 loading，避免重复提交。

### 5.5 后台与表格

后台页面不在 Phase 7 完全卡片化。桌面端保留表格；移动端重点保证不撑破页面：表格外层可横向滚动，筛选和操作按钮折行，重要指标卡片改为两列或一列。

### 5.6 可访问性基础

关键按钮必须有明确文本，不只依赖图标；弹窗可通过键盘关闭，表单控件有可见 label；低对比辅助文案使用统一灰度；移动端按钮点击区域不小于常规 Element Plus 按钮尺寸；错误信息不要只用颜色表达。

## 6. 技术方案

主要修改 `frontend/src/styles.css`、`frontend/src/layouts/MainLayout.vue` 和现有视图文件。允许新增少量组件：`EmptyState.vue`、`FormSection.vue`、`PageActions.vue`。组件数量控制在 2-4 个以内，避免演变成设计系统重写。

Phase 7 默认不改后端、不新增 Flyway 迁移。若发现某个页面缺少必要错误信息，优先在前端显示现有 API 错误。

Phase 7 只做 bundle review，不把 Element Plus 按需导入或路由级拆包作为必做项。若 Vite large chunk warning 仍只是已知警告，不阻塞发布。

## 7. 实施顺序

1. 建立共享响应式基础：全局布局、移动菜单、页面容器、状态卡片、操作区和表单分区样式。
2. 打磨登录注册、通知、身份保证金、钱包、信用中心等账户基础页。
3. 打磨跑腿任务大厅和任务工作台。
4. 打磨二手商品列表、详情和发布页。
5. 打磨学生店铺列表、详情和商家工作台。
6. 打磨项目广告列表、详情和管理页。
7. 打磨后台运营、治理和审核页的移动可读性。
8. README/CLAUDE handoff，服务器 Docker build，API smoke，Playwriter 桌面/平板/手机验证。

## 8. 验证计划

完整构建优先通过服务器低影响 Docker build 或用户明确允许的环境执行。API smoke 覆盖 `/api/goods`、`/api/tasks`、`/api/shops`、`/api/project-ads`、`/api/admin/ops/analytics/overview`、`/api/admin/governance/dashboard`、`/api/credit/users/1`。

Browser / Playwriter 验证视口：桌面 1365x768、平板-ish 820x1180、手机 390x844。重点路由覆盖用户侧和管理侧主要页面。每个页面检查：无白屏、无明显 Element Plus 错误、无横向溢出、主按钮可见、表单/弹窗可操作、空状态可理解。

## 9. 非目标

Phase 7 不做原生移动 App、微信小程序、全量设计系统重写、替换 Element Plus、支付中心接入强化、JWT/RBAC 全量硬化、新业务线或新数据库表、实时聊天、地图选点、复杂推荐、每笔订单资金托管或逐单保证金冻结。

## 10. 自检

范围聚焦在响应式 Web 与体验一致性，没有引入新业务闭环。技术方案沿用 Vue 3、Element Plus 和现有集中 CSS，避免大规模重构。验证计划覆盖桌面、平板-ish 和 390x844 移动视口。支付、隐私、迁移和服务器低影响约束均保持不变。
